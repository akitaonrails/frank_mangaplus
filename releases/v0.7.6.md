## FRANK MANGA+ v0.7.6

Second-stage Wayland fix — addresses [#1](https://github.com/akitaonrails/frank_mangaplus/issues/1).

### What was broken

v0.5.2 added `WEBKIT_DISABLE_DMABUF_RENDERER=1` to bypass the `Could not create GBM EGL display: EGL_SUCCESS. Aborting…` crash on Wayland. That covered the GBM/DMABUF code path inside WebKitGTK.

A user reported a *different* EGL failure on Bazzite (Fedora atomic) + AMD Lucienne integrated GPU + KWin Wayland:

```
Could not create default EGL display: EGL_BAD_PARAMETER. Aborting...
```

That's the SHM (software-mapped) compositing path failing as well. The window mounts (title bar appears), the WebView aborts before painting anything, you see a blank white panel.

### What changed

The binary now also sets `WEBKIT_DISABLE_COMPOSITING_MODE=1` at startup, pushing WebKit into full CPU-side compositing — no EGL at all. Combined with the existing DMABUF disable, this covers both reported EGL initialisation failures.

Performance impact for our use case is negligible: manga pages are mostly static; the page-flip animation and CSS sepia filter run on the CPU without any visible drop. Browsers do *much* heavier compositing — a static image viewer is roughly free.

Both env vars are still set conditionally — if you've already set either yourself (`export WEBKIT_DISABLE_COMPOSITING_MODE=0` to force hardware compositing back on, say), the binary respects your value.

### If you still see EGL crashes

A few escalations worth trying as env vars before launch:

- `WEBKIT_FORCE_SANDBOX=0` — disables WebKit's process sandbox (some distros block EGL there)
- `GDK_BACKEND=x11` — forces XWayland; EGL sometimes works on the X path when native Wayland doesn't
- `LIBGL_ALWAYS_SOFTWARE=1` — last resort, full software rendering of the entire GL stack

If none of those work, file an issue with `inxi -G` (or `fastfetch`) output and the AppImage launch logs.
