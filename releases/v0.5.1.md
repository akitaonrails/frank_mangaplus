## FRANK MANGA+ v0.5.1

AUR-only fix. The `.deb`, `.rpm`, `.AppImage`, `.dmg`, and Windows installer behave identically to v0.5.0.

### What was broken

The AUR `PKGBUILD` looked for the bundle's `.desktop` file at a hardcoded path: `usr/share/applications/mangaplus-desktop.desktop`. But Tauri names the file after the product name (`FRANK MANGA+.desktop`), and has done so since v0.4.0. So the conditional that copies the file into the package silently failed every time — the .desktop entry never made it into AUR installs, and the launcher never showed up in any menu, including the .desktop improvements that v0.5.0 was supposed to ship.

### What changed

In `aur-publish.yml`'s PKGBUILD template:

- The `.desktop` file is now found by glob (`find ... -name '*.desktop'`) rather than by hardcoded name.
- The package installs **two** `/usr/bin` symlinks: `mangaplus-desktop` (so the launcher's `Exec=` and `StartupWMClass=` lines resolve) and `mangaplus-reader` (kept as the familiar alias).
- Icons stay named `mangaplus-desktop.png` to match the bundle's `Icon=` reference, no rewriting.

`yay -Syu` will pull this and you'll see FRANK MANGA+ in your launcher under Graphics, searchable by `manga` / `comic` / `reader` etc.
