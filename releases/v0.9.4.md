## FRANK MANGA+ v0.9.4

Full-catalog search via a stale-while-revalidate cache; fast thumbnail loading; library-state-aware "+ Library" button. Big internal cleanup pass — every new code path was audited and hardened before shipping.

### Search

- The search page now uses the full MANGA Plus catalog (~thousands of titles, fetched from `title_list/all_v3`) rather than only the curated ~531 in `title_list/search`. Titles like "Bleach" that exist in the catalog but not in the curated list now turn up when you type.
- The curated catalog still loads on mount for the fast landing view; the full catalog is lazy-fetched the first time you type a character. Subsequent keystrokes filter against it instantly.
- Results capped at 80 visible with a "Showing first N of M — refine your query" footer.
- "+ Library" button now reflects state: `+ Library`, `Adding…`, `✓ In Library`, or `Retry`. Buttons disable while in-flight or already added.

### Cache (`~/.cache/mangaplus-reader/all_titles_<lang>_<clang>.bin`)

- Stale-while-revalidate. First read: blocks on network. Within 24h: serves from disk instantly. After 24h: still serves from disk *and* kicks off a background refresh that emits an `all_titles_refreshed` Tauri event when fresh data lands.
- TTL configurable via `MANGAPLUS_CATALOG_TTL_HOURS=N`.
- Atomic writes via `*.tmp` + rename — a crash mid-write never leaves the bin/meta pair inconsistent.
- Per-locale dogpile guard so 10 simultaneous stale reads spawn one refresh, not ten.
- Decode failures (old shape, truncated, corrupt) evict the bad files and fall through to a fresh fetch — no `rm -rf` required by the user.
- Filter applied on both write *and* read paths so caches written by older builds correct themselves transparently.

### Image fetching (thumbnails + chapter pages)

- The previous behaviour serialized **every** image fetch behind the 500ms API rate-limit mutex — a cold-cache 80-thumbnail grid took 40 seconds to populate. The API mutex is now scoped to actual API calls; CDN fetches use a separate 12-permit semaphore for genuine parallelism.
- Cache-hit fast path skips the semaphore entirely — re-renders never queue behind in-flight network fetches.
- Per-image retry with exponential backoff: 250ms → 500ms → 1s on transient failures (timeout, connection reset, 408, 429, 5xx). Hard 4xx (404, 403…) fail fast.
- **Critical fix from the audit:** backoff sleeps now happen *without* holding the semaphore permit. The earlier version pinned a permit for the full ~1.75s backoff window, collapsing effective concurrency under retries.
- Image body size cap: 32 MB. Defends against a misbehaving CDN streaming gigabytes.

### Robustness — what the pre-release audit caught

A three-agent code review surfaced 14 issues. The 8 most impactful were fixed before shipping; the other 6 were addressed as a "make it robust" pass after the user asked for nothing-deferred.

- **Partial-success on `tokio::join!`**: a 429 on either of the two catalog buckets no longer loses the whole refresh. Whatever succeeded is persisted; both-failed surfaces the error.
- **Permit released before backoff sleep** (see above).
- **Cache decode fall-through**: a corrupt cache evicts itself and triggers a network refetch instead of stuck-with-no-recovery.
- **Locale matching captured in component state**, not module constants — a future language switcher won't drop in-flight refresh events.
- **Unknown-language canary**: a one-time stderr warn fires the first time MANGA Plus ships a new language enum value we haven't mapped, so the new titles don't silently disappear from filtered results.
- **Case-insensitive locale matching**: `"ESP"` no longer silently falls through to English titles.
- **Refresh-skipped log** when the dogpile guard blocks a duplicate refresh — visibility into a previously silent path.

### Backwards compatibility

- Cache files written by any prior build (which decoded into a different shape) are now evicted gracefully on first read; you don't need to delete anything by hand.
- The `lang_str_to_enum` table mirrors `src/lib/lang.ts` exactly — adding a new language requires a one-line addition on both sides (caught by a canary log line in production).
- No changes to the v0.9.0 render-mode policy or the v0.9.3 page-sizing.
