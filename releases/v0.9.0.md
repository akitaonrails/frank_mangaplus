## FRANK MANGA+ v0.9.0

Linux render policy + auto-recovery. **Most users get GPU acceleration back.**

### What changed

Through v0.8.0 the binary unconditionally set `WEBKIT_DISABLE_COMPOSITING_MODE=1` on Linux — full CPU rendering for everyone, just to keep the ~5% of users on broken EGL setups working. v0.9.0 replaces that hammer with a tiered policy that picks the lightest workaround for your actual hardware, and auto-recovers if it gets the wrong answer.

### The new policy

On every Linux launch, the binary now reads:

1. `MANGAPLUS_RENDER_MODE` env var (user override)
2. `~/.config/mangaplus-reader/render.conf` (`mode = ...`, persistent override)
3. `~/.config/mangaplus-reader/render-recovery` marker (previous launch died?)
4. GPU vendor from `/sys/class/drm/card*/device/vendor`
5. Display server from `$WAYLAND_DISPLAY` / `$DISPLAY`

…then picks one of four modes:

| Mode | Env vars set | When auto-detect picks it |
|---|---|---|
| `native` | none — full GPU, WebKit defaults | X11 sessions |
| `dmabuf-off` | `WEBKIT_DISABLE_DMABUF_RENDERER=1` | Wayland + AMD or Intel (SHM compositing still GPU-accelerated) |
| `nvidia-light` | + `__NV_DISABLE_EXPLICIT_SYNC=1` | Wayland + NVIDIA |
| `safe` | + `WEBKIT_DISABLE_COMPOSITING_MODE=1` | Crash recovery (last launch failed) |

Whatever it applied lives in `~/.config/mangaplus-reader/render-state.log` — `cat` that file to see exactly what your binary is doing.

### Auto-recovery

Startup touches the recovery marker; the frontend deletes it on first successful render via a new `mark_app_ready` Tauri command. If the WebView aborts before that fires (the classic blank-white-screen EGL crash), the marker stays behind. The **next** launch reads the marker → falls back to `safe` mode automatically → working app, no docs needed.

This means a Bazzite + AMD + KWin Wayland user (issue [#1](https://github.com/akitaonrails/frank_mangaplus/issues/1)) hitting `EGL_BAD_PARAMETER` on first launch now gets a working app on the second launch with no intervention.

### Manual overrides

Quick test (single launch):

```bash
MANGAPLUS_RENDER_MODE=safe ./FRANK.MANGA+_*_amd64.AppImage
# also: native, dmabuf-off, nvidia-light, auto
```

Persistent — `~/.config/mangaplus-reader/render.conf`:

```
mode = nvidia-light
```

The env var wins if both are set.

### New `docs/troubleshooting.md`

Walks through every mode, the auto-recovery loop, common scenarios (blank screen, sluggish flip, "worked yesterday"), and the last-resort escape hatches (`GDK_BACKEND=x11`, `WEBKIT_FORCE_SANDBOX=0`, `LIBGL_ALWAYS_SOFTWARE=1`). Linked from the README header nav and the install section.

### Tests + code structure

Everything decision-making lives in `src-tauri/src/render_env.rs` as pure functions; `lib.rs::run()` is just side-effecting wiring. The integration is small enough to inspect at a glance.

**25 new Rust tests** covering:
- `parse_vendor_hex` across casing/prefix variants
- `RenderMode::env_vars` layering
- `decide_mode` for every `(display, gpu)` combo + explicit override + recovery precedence
- `resolve_user_override` (env wins over config, garbage fall-through, neither set)
- Recovery marker lifecycle (create / clear / idempotent)
- `parse_config_mode` with comments, quotes, multi-key files
- `write_state_log` content shape for both Native (no env) and NvidiaLight (env listed)

**29 desktop crate tests total** (was 4 before this batch). Frontend vitest unchanged at 49. Clippy clean on `--lib --tests` for both crates.

### Backwards compatibility

If you had a working v0.8.0 install, **nothing changes for you** unless you're on Linux. Linux users on:

- **X11**: get GPU acceleration back (was force-disabled in v0.8.0)
- **Wayland + AMD / Intel**: get GPU compositing back via SHM
- **Wayland + NVIDIA**: get GPU compositing back via the explicit-sync workaround
- **Bazzite-style broken EGL combos**: still work, via auto-recovery → safe mode on the second launch

If anything regresses for you on Linux, set `MANGAPLUS_RENDER_MODE=safe` to get the v0.8.0 behaviour back, and please file an issue with your `render-state.log` contents.
