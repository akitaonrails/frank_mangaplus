## FRANK MANGA+ v0.7.3

Real fix for the page indicator on chapter transitions.

### What v0.7.2 still got wrong

v0.7.2 computed the current chapter's first index by `findIndex` over `loadedPages` with `p.chapterId === visibleChapterId`. The intent was right: scope the indicator to the chapter the user is reading. But `visibleChapterId` and `loadedPages` are both derived from reactive state, and during Svelte's update pass they can momentarily disagree. When that happened, `findIndex` returned -1, `Math.max(0, -1)` silently masked it as 0, and the indicator reported `currentPageIndex + 1` instead of the chapter-local page number.

Confirmed by probing Kaiju No. 8 around the **ex → #077** boundary:

```
[91] id=1015153 name="ex"    ← in loadedPages at indices 0..9
[92] id=1015155 name="#077"  ← in loadedPages at indices 10..28
```

Crossing into #077 should show "1 / 19". v0.7.2 showed "11 / 19" — exactly `currentPageIndex (=10) + 1`. The header chapter name correctly switched to "#077", because that derivation reads from `loadedPages[currentPageIndex].chapterName` and doesn't take the findIndex code path.

### What changed

Chapter bounds are now computed by scanning backward/forward from `currentPageIndex` directly, since chapter pages are always contiguous (we `appendChapter()` whole chapters at a time and never insert anywhere but the end):

- `currentChapterFirstIndex` walks left while the previous page shares the same `chapterId`.
- `chapterPageCount` walks right while the next page shares the same `chapterId`.

Both derive from a single source (`loadedPages[currentPageIndex]`), removing the chance of a stale `visibleChapterId` desyncing from the array.

### Why probing mattered

Looked at the actual API response for ex / #075 / #076 / #077 / #078 — each `manga_viewer_v3` correctly stamps its own `chapter_id` on its pages, no cross-contamination. That ruled out a server-side data bug and pointed straight at the reactive-update timing on the client.
