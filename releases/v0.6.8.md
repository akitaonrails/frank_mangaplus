## FRANK MANGA+ v0.6.8

Follow-up to v0.6.7's chapter-list fix.

### What was still broken

v0.6.7 swapped `viewer.chapters` (3 entries, truncated) for `title_detail.chapter_list_v2` (canonical full list, 150 entries for Kaiju No. 8) — but it still sorted that list by `chapterId` ascending. Live probe against Kaiju surfaced a problem with that sort:

```
sorted by chapter_id, around the boundary:
  id=1023544 name="#123"
  id=1023808 name="#125"    ← lower id, higher chapter number
  id=1024011 name="#124"    ← higher id, lower chapter number
  id=1024314 name="#126"
```

Chapter 125 has a *lower* `chapterId` than chapter 124. Probably because the API reassigned ids during a re-upload at some point. The reading order is 123 → 124 → 125 → 126, but the id ordering says 123 → 125 → 124 → 126.

So v0.6.7 would have unblocked your specific chapter 54 → 55 navigation (whose ids *are* monotonic), but for Kaiju readers around chapter 124–125 the auto-advance would still send you into the wrong chapter.

### What changed

The canonical `chapter_list_v2` arrives in publication order already (that's how `title_detail` returns it). v0.6.8 drops the sort and preserves the natural order. Auto-advance now visits chapters in the order Shueisha published them, regardless of internal id reshuffles.

The viewer.chapters fallback (still used for the first ~1 s before `title_detail` arrives) is still sorted by id — it's only ever the first three chapters of the title, where ids and numbers are monotonic.
